# Product Requirements Document: Food Preferences Survey

## Overview

Build a short web-based survey targeted at college students that collects information about their food preferences and eating habits. The app has two pages: a survey form and a results page showing aggregated responses. Responses are stored in a Supabase PostgreSQL database.

---

## Tech Stack

- **Framework:** Vite + React
- **Database:** Supabase (PostgreSQL)
- **Hosting:** Azure Static Web Apps
- **Charting:** Recharts

---

## Survey Questions

The survey must include the following four questions, using the specified input types:

| #   | Question                                   | Input Type                          |
| --- | ------------------------------------------ | ----------------------------------- |
| 1   | What is your favorite type of food?        | Text input box                      |
| 2   | What meal do you eat most often?           | Dropdown list                       |
| 3   | How often do you eat takeout each week?    | Radio buttons                       |
| 4   | What foods do you usually enjoy?           | Checkboxes + conditional text input |

### Question Details

**Q1 — Favorite Food**

- Free-text input
- Placeholder: "e.g. Italian, Mexican"
- Autofocus
- Required field

**Q2 — Most Common Meal**

- Dropdown list with:
  - Breakfast
  - Lunch
  - Dinner
  - Snacks
- Required field

**Q3 — Takeout Frequency**

- Radio button group with:
  - Never
  - 1–2 times per week
  - 3–4 times per week
  - 5+ times per week
- Required field

**Q4 — Foods You Enjoy (select all that apply)**

- Checkboxes:
  - Pizza
  - Pasta
  - Burgers
  - Salad
  - Tacos
  - Other
- If "Other" is checked, show a text input box
- At least one must be selected (required)
- "Other" text box is required if selected

## Form Behavior

- All fields are required; display inline validation error messages if the user submits without completing a field
- The "Other" food text input must be keyboard accessible when it appears dynamically — focus should move to it automatically when "Other" is checked
- On successful submission, display a thank-you confirmation screen showing a summary of the user's answers
- The thank-you screen includes two buttons:
  - "View Results" — navigates to the results page
- The submit button shows "Submitting..." while the request is in flight
- If the database insert fails, display an error message and keep the form intact

---

## Results Page

A separate `/results` route displays aggregated, anonymized data from all submissions. Individual responses are never shown — only group-level summaries.

### Charts / Visualizations

1. **Total Responses** — a large number prominently displayed at the top
2. **Takeout Frequency** — vertical bar chart, one bar per option, ordered from Never to 5+ times per week
3. **Most Popular Foods** — horizontal bar chart, sorted by count descending. For respondents who selected "Other", use the text they entered in `other_food` as the food label rather than displaying "Other". Normalize casing before counting so similar answers are treated as the same entry. These user-entered foods appear as individual bars alongside the standard food options.
4. **Most Common Meal** — horizontal bar chart showing counts for Breakfast, Lunch, Dinner, and Snacks

Use a charting library (e.g. Recharts) for all visualizations.

### Navigation

- Home page has a short welcome message with a "Take the Survey" button linking to the survey form and a "View Results" button linking to the results page
- The results page has a "Home" button linking back to the home page
- The survey form has a "View Results" button in the header linking to the results page

### Environment Variables

The app connects to Supabase using two environment variables:

| Variable                 | Description                                                 |
| ------------------------ | ----------------------------------------------------------- |
| `VITE_SUPABASE_URL`      | Your Supabase project URL (e.g. `https://xxxx.supabase.co`) |
| `VITE_SUPABASE_ANON_KEY` | Your Supabase anon/public key                               |

Both values are found in the Supabase dashboard under **Project Settings → API**.

- For local development, create a `.env` file in the project root with these values. Add `.env` to `.gitignore`.
- For production, add both as GitHub Actions secrets. The CI/CD workflow injects them at build time.

## Design & Quality Standards

### Appearance

- Simple, minimalist design throughout
- White or light neutral background with dark body text
- `#8A3BDB` is the primary accent color — use it for buttons, links, active/focus states, borders, and chart colors
- Do not use `#8A3BDB` for body text; it does not meet WCAG AA contrast requirements at small sizes. Use a dark neutral (e.g. `#1a1a1a`) for body text
- For any text rendered in the accent color (e.g. headings or labels), verify contrast passes WCAG AA before using it

### Responsive Layout

The app must be usable on mobile, tablet, and desktop screen sizes. Use a single-column layout on small screens. Do not assume desktop-only use — the survey will likely be completed on phones.

### HTML5 Validation

All pages must validate as HTML5 with no errors using the [W3C Markup Validation Service](https://validator.w3.org/).

### Accessibility — WCAG 2.1 Level AA

All pages must pass WCAG 2.1 Level AA. Key requirements:

- All form inputs must have a programmatically associated `<label>` element
- Inline validation error messages must be programmatically associated with their fields using `aria-describedby`
- The "Other" food text input must be keyboard accessible when it appears dynamically — focus should move to it automatically when "Other" is checked
- All images and icons must have appropriate `alt` text
- Color alone must not be used to convey meaning (e.g. error states must include text, not just a red border)
- All interactive elements must be reachable and operable via keyboard
- Use sufficient color contrast for all text — minimum 4.5:1 for normal text, 3:1 for large text

## Deployment

This app is deployed as an **Azure Static Web App** using Vite + React.

Because the app uses client-side routing (`/` and `/results`), a `staticwebapp.config.json` file must be included in the `public/` directory so Azure serves `index.html` for all routes instead of returning 404s on direct navigation or page refresh.

```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/assets/*"]
  }
}
```

This file is automatically copied to `dist/` during the Vite build.
